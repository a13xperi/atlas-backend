## 5. Security Findings

### CRITICAL: Hardcoded Supabase Anon Key in Source Code

**File:** `services/api/src/routes/qa.ts:10`

```typescript
const QA_SUPABASE_KEY = process.env.QA_SUPABASE_KEY || "eyJhbGciOiJIUzI1NiIs...";
```

A Supabase anon key is hardcoded as a default value. While anon keys have limited permissions (RLS-enforced), this key is committed to the Git repo and grants read/write to any table with permissive RLS policies. The key belongs to Supabase project `zoirudjyqfqvpxsrxepr`.

**Recommendation:** Move to Railway env vars. Remove the hardcoded default. If this is intentionally the anon key with RLS protection, add a code comment documenting that decision.

### MEDIUM: Dev-Only JWT Secret Fallback

**File:** `services/api/src/lib/config.ts:58`

```typescript
JWT_SECRET: process.env.JWT_SECRET || "dev-only-secret-do-not-use-in-production"
```

The fallback is only used when Zod validation fails in non-production. In production, the app exits if `JWT_SECRET` is missing. This is correctly implemented but the fallback string is predictable -- any local dev instance shares the same JWT secret.

**Recommendation:** No action needed for production. For local dev, consider using a randomly generated secret per `npm install` (e.g., via a postinstall script).

### LOW: GitHub Token Not Validated at Startup

**File:** `services/api/src/routes/loop.ts:63`

The `GITHUB_TOKEN` is checked at request time (returns 503 if missing). This is fine for an optional feature, but means PR creation silently fails until someone hits the endpoint.

## 6. Undocumented Variables Summary

The following 12 env vars are used in code but NOT in `.env.example`:

1. `GOOGLE_AI_API_KEY` -- Gemini AI provider (images, generation)
2. `GEMINI_MODEL` -- Model override (default: gemini-2.5-flash)
3. `XAI_API_KEY` -- Grok AI provider
4. `TWITTER_BEARER_TOKEN` -- X/Twitter API read access
5. `TWITTER_CLIENT_ID` -- X OAuth2 app ID
6. `TWITTER_CLIENT_SECRET` -- X OAuth2 app secret
7. `TWITTER_OAUTH_CALLBACK_URL` -- X OAuth2 redirect URI
8. `SENTRY_DSN` -- Error monitoring
9. `QA_SUPABASE_URL` -- QA test run storage (separate Supabase project)
10. `QA_SUPABASE_KEY` -- QA test run storage key
11. `GITHUB_TOKEN` -- GitHub API for PR creation from loop
12. `GITHUB_OWNER` -- GitHub repo owner for loop PR creation
13. `GITHUB_REPO` -- GitHub repo name for loop PR creation
14. `RALPH_STATE_PATH` -- Filesystem path for AutoResearch loop state
15. `BACKUP_RETENTION_DAYS` -- R2 backup retention (script only)

## 7. Recommendations

### Immediate (before next deploy)

1. **Add undocumented vars to `.env.example`** -- All 15 vars listed above should be documented with placeholder values and comments explaining which feature they enable.

2. **Remove hardcoded Supabase key from qa.ts** -- Replace with env var only, no fallback. The route should return 503 if the QA Supabase is not configured.

### Short-term

3. **Move QA/loop vars into centralized config** -- `QA_SUPABASE_URL`, `QA_SUPABASE_KEY`, `GITHUB_TOKEN`, `GITHUB_OWNER`, `GITHUB_REPO`, and `RALPH_STATE_PATH` should be added to the Zod schema in `lib/config.ts` for consistent validation.

4. **Add Railway env var checklist** -- Document which vars Railway auto-provides (`DATABASE_URL`, `REDIS_URL`, `PORT`) vs. which must be manually set in Railway dashboard.

5. **Verify Railway has all required vars** -- Run `railway variables` or check the Railway dashboard to confirm these are set in production:
   - `JWT_SECRET`
   - `FRONTEND_URL` (should include production + staging origins)
   - `GOOGLE_AI_API_KEY`
   - `XAI_API_KEY`
   - `ANTHROPIC_API_KEY`
   - `OPENAI_API_KEY`
   - `TWITTER_BEARER_TOKEN`, `TWITTER_CLIENT_ID`, `TWITTER_CLIENT_SECRET`, `TWITTER_OAUTH_CALLBACK_URL`
   - `TELEGRAM_BOT_TOKEN`
   - `SENTRY_DSN`
   - `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_JWT_SECRET`

### Nice-to-have

6. **Add startup banner** -- Log which optional features are enabled/disabled based on which keys are present (the smoke-test script already does this pattern well).

7. **Telegram bot service uses `BOT_TOKEN`** instead of `TELEGRAM_BOT_TOKEN` -- If the telegram-bot service is ever deployed, unify the naming.
